MergeLife Cross-Engine Conformance Vectors
==========================================

This repository ships four independent implementations of the MergeLife
update rule (Python, JavaScript, Java, C). They are required to produce
**byte-identical lattices**. The vectors in [vectors.txt](vectors.txt) pin
that requirement down: every language's test suite replays them, so an engine
that drifts from the paper fails its own tests immediately.

History note: the algorithms diverged silently once before — the shipped
macOS app carried a pre-paper-correction engine for over two years because
nothing enforced agreement.

The contract
------------

Each line of `vectors.txt` is:

    rule rows cols seed steps fnv1a64

An engine passes a vector when the following procedure reproduces the hash:

1. **Initial lattice.** Fill a `rows x cols` RGB lattice from the LCG below,
   iterating row-major, channels innermost (`for row / for col / for R,G,B`).
   Only the *current* lattice is specified; engines with a back buffer may
   leave it zeroed (since the paper correction, unmatched cells are seeded
   from the current lattice, so the back buffer never leaks into results).
2. **Run** exactly `steps` update steps.
3. **Hash** the resulting current lattice in the same row-major RGB byte
   order with FNV-1a 64 and compare against `fnv1a64` (16 lowercase hex
   digits).

### PRNG (initial lattice)

32-bit linear congruential generator, one byte per advance:

    state = seed                                  (unsigned 32-bit)
    next byte:  state = (state * 1664525 + 1013904223) mod 2^32
                value = state >> 24

### Hash

FNV-1a, 64-bit:

    h = 0xcbf29ce484222325
    per byte:  h = (h XOR byte) * 0x100000001b3   mod 2^64

Both algorithms are implemented *independently in each language's test
harness* (about ten lines each); only the engine under test is shared code.

Regenerating
------------

Vectors are generated by the reference Python engine — the mergelife library
([python/mergelife-lib](../python/mergelife-lib)):

    python conformance/gen_vectors.py

The script uses the installed `mergelife` package if present and otherwise
falls back to the in-repo library source (numpy/scipy still required).
Regeneration is deterministic; `git diff` after running it should be empty
unless the algorithm itself changed. If you change the algorithm on purpose,
regenerate, and expect every other engine to fail until it is updated to
match.

Where the tests live
--------------------

| Engine | Test |
|---|---|
| Python (mergelife library) | `python/mergelife-lib/tests/test_conformance.py` |
| JavaScript | `js/mergelife-lib/test/test-conformance.js` |
| Java | `java/evolve/src/test/java/org/heatonresearch/util/TestConformance.java` |
| C | `c/test/conformance.c` (`make test`) |

The PyQt app has no engine of its own; it imports the mergelife library, so
the library's conformance test covers it.
